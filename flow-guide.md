# MVP Flow — 진행 가이드 (템플릿)

**디자이너는 채팅만 보면 됩니다.** 다음에 할 일은 제가 말해 드리고, **제가 "○○ 파일 열어서 여기 복사해줘"라고 할 때만** 그 파일을 열어서 복사하시면 됩니다. 파일 탐색은 제가 시키는 것만 하면 됩니다.

**이 파일(flow-guide)은 덮어쓰지 않습니다.** 이번 프로젝트의 실제 값·산출물은 모두 **`project/`** 폴더에만 채웁니다. 진행이 궁금하면 채팅에 **"지금 어디야?"** 또는 **"다음 액션 알려줘"**라고만 하세요. 제가 현재 단계를 알려드립니다.

---

## 원칙 — 데이터 소스·호출 최소화

아래 원칙은 제가 기획·구현 방향을 잡을 때 참고하는 것이에요. **디자이너는 읽지 않으셔도 됩니다.**

**1. 외부 API 호출량을 최대한 줄인다.**  
화면을 열 때마다 API를 호출할 필요는 없다. 검색 조건(도시, 날짜 등)과 결과를 **Supabase 등 DB에 저장**하고 **캐시 유효 시간**을 두면, 유효 시간 내 같은 요청은 DB에서 먼저 제공할 수 있어 API 호출이 크게 줄어든다.  
PRD 작성·구현 시 이 원칙을 반영한다 (캐시 전략, 저장 구조, TTL 명시).

**2. 필요하면 외부 API뿐 아니라, 합법적인 외부 사이트 크롤링을 고려한다.**  
API가 없거나 비용·제한이 클 때, 해당 사이트의 **robots.txt·이용약관**을 확인한 뒤 크롤링을 옵션으로 검토한다.

**3. 외부 크롤링 시, 서비스 성격에 따라 크롤링 주기를 세팅하고, 주기적으로 크롤링하여 최신성을 유지하는 방향을 고민한다.**  
한 번만 수집하지 않고, 일/주/월 단위 등으로 갱신하고 결과를 DB에 저장해 두면, 호출 부담과 최신성 사이의 균형을 잡을 수 있다.

## 0. 시작 전 준비 (필요한 서비스)

이 플로우를 따라가려면 **아래 네 서비스**를 쓰게 됩니다. 개발 경험이 없어도 괜찮도록, **가입만 미리 해 두시면** 1·7·8단계에서 편합니다. 프로젝트나 저장소(레포)는 해당 단계에서 만들면 됩니다.

| 서비스 | 한 줄 설명 | 미리 할 일 |
|--------|------------|------------|
| **GitHub** | 코드 저장소. 1번에서 Vercel이 여기 연동합니다. | [가입](https://github.com/join) |
| **Vercel** | 웹 앱 배포. 1번에서 템플릿 배포로 저장소(레포)·배포 URL이 생깁니다. | [가입](https://vercel.com/signup) (GitHub로 로그인 가능) |
| **Jules** | 구현을 도와주는 AI. 8·9번에서 프롬프트 붙여넣고 수정 제안(PR)을 받습니다. | [가입·접속](https://jules.vercel.app) (GitHub 연동) |
| **Supabase** | DB(데이터 저장). 7번에서 프로젝트 만들고 SQL 실행합니다. | [가입](https://supabase.com/dashboard) (7번 전에 프로젝트 생성) |

처음이시면 **"시작해줘"**라고만 하시면, 제가 init부터 1번까지 순서대로 안내합니다.

**이 가이드에서 자주 쓰는 말**  
- **PR**: 코드 수정 제안서. Jules가 올리면 GitHub에서 확인할 수 있어요.  
- **Merge**: 그 수정안을 반영(합치기)하는 것.  
- **Preview URL**: 수정본 미리보기 링크. 배포 전에 확인하는 주소.  
- **레포(저장소)**: GitHub에 만드는 프로젝트 저장 공간.  

자세한 건 맨 아래 **용어 한줄 정리**를 참고하세요.

## 1. 먼저 할 일

**이걸 해야 2번 이후가 모두 가능해요.**

**한 줄 요약**: [이 링크](https://vercel.com/templates/next.js/nextjs-boilerplate)를 누르고, 화면에 나오는 대로 **Deploy → 다음 → 다음** 하면 끝입니다. GitHub 연동·저장소(레포) 이름 입력만 따라 하시면 **GitHub URL**과 **Vercel URL**이 생깁니다.

- [Next.js Boilerplate](https://vercel.com/templates/next.js/nextjs-boilerplate) 템플릿 페이지로 이동
- **Deploy** 클릭 → Vercel이 GitHub 연동 안내 → GitHub 권한 승인(처음 1회) → 저장소(레포) 이름(Repository Name) 입력(예: `meet-labs-web`) → **Continue / Deploy**
- 완료되면 Vercel에서 배포된 웹 주소(Production URL, `*.vercel.app`) 표시, GitHub에 새 저장소(레포) 생성됨
- **막히면** 화면 캡처를 채팅에 보내 주세요. 제가 다음에 뭘 누르면 되는지 안내합니다.

**TIP — 어디를 보면 돼요?**  
배포가 끝나면 **Vercel 화면**에 "Congratulations"와 함께 **Visit** 링크(배포 URL)가 보여요. GitHub는 **Vercel 대시보드 → 해당 프로젝트 → Settings → Git**에서 연결된 Repository 이름을 누르면 저장소 주소로 갑니다. 또는 **GitHub 로그인 후 Your repositories**에서 방금 생긴 레포를 찾으면 됩니다.

**입력 보관**: `project/info.md`

## 2. 프로젝트 정보 (1번 완료 후)

프로젝트명, GitHub URL, Vercel URL을 알려주세요.

**TIP — 어디를 보면 돼요?**

- **Vercel URL**: Vercel 로그인 → **Dashboard** → 방금 배포한 프로젝트 클릭 → **Domains** 또는 상단 **Visit** 링크가 배포 URL (예: `https://레포이름.vercel.app`)

**입력 보관**: `project/info.md`

## 3. 프로젝트 스펙 정의 (시안 이미지 필수)

**해 주실 일**
- **프로젝트 기획서를 전달해 주세요.** 간단한 화면이면 한두 줄 설명만이라도 좋아요. 화면을 개발해야 하니 **Figma나 Stitch 시안은 필수**예요.
- 전달해 주시면 제가 그걸 바탕으로 상세 스펙을 정리해 `project/screens/`와 `project/screens/README.md`에 채웁니다.

**TIP — 어디를 보면 돼요?**  
Figma/Stitch에서 **화면 캡처**나 **Export → PNG**로 내려받은 이미지를 채팅에 붙여넣으면 됩니다. 여러 장이면 한 장씩 첨부하고, "이 화면은 ○○ 화면이에요"처럼 한두 줄만 설명해 주세요. 시안을 그리기 어려우면 [Stitch](https://stitch.withgoogle.com/)를 추천할게요!

**입력 보관**: `project/screens/`, `project/screens/README.md`

## 4. Plan 생성 (및 검토)

3번에서 기획서·시안을 받은 뒤, 디자이너에게 **"기획서와 시안을 모두 제공해 주셨을까요? 모두 제공해 주셨다면 제가 기획서와 시안을 기반으로 계획을 세워 드리겠습니다."**라고 안내한 뒤, 제공이 완료되었으면 기획서·시안을 바탕으로 **A–Z 할 일**을 정리한 Plan/PRD를 생성합니다.

**PRD 작성 시 두 관점을 반영한다.**  
- **시니어 기획자 관점**: 요구사항·화면 스펙·예외 시나리오·우선순위·사용자 메시지(에러·빈 상태 등)를 명확히 정리한다.  
- **시니어 개발자 관점**: 데이터가 외부 API·크롤링일 경우 **구현 레벨 리서치**를 한다. 필요하면 웹 검색으로 후보를 조사한 뒤, PRD의 **데이터 전략** 섹션에 **(1) 후보 소스 2~3개 이상 구체 명시** (예: KRX OOO, 네이버 금융 XXX 페이지, OO 공개 API), **(2) 각 옵션의 난이도·리스크·이유** (이용약관, robots.txt, 동적 렌더링, 로그인 필요, 빈도 제한 등), **(3) 권장안과 이유**를 적는다. 크롤링이 어렵다면 **왜 어려운지** (정책·기술·운영 중 어떤 이유인지) 반드시 한 줄이라도 명시한다.

생성 후 **"project/PRD.md를 한 번 확인해 보시고, 수정하거나 더 넣고 싶은 게 있으면 말해 주세요. 괜찮으시면 다음은 5번 필요한 정보 수집이에요."**라고 안내한다. 디자이너가 질문·수정을 요청하면 반영하고, **OK** 할 때까지 4번 단계이다. **OK** 하면 5번으로 넘어간다.

**산출물**: `project/PRD.md`

## 7. Supabase 세팅

**한 줄 요약**: 제가 만들어 준 SQL 문장을 **Supabase SQL Editor**에 붙여넣고 **Run** 한 번 누르면 됩니다. 그다음 **RLS**(읽기/쓰기 허용 설정)만 켜 주고, **Project URL**과 **anon key**를 채팅으로 알려 주시면 제가 `project/info.md`에 반영합니다. Vercel 환경변수에도 동일하게 넣어 두세요.

- Supabase 프로젝트 생성 → **SQL Editor**에서 `project/db-design.md`의 SQL 실행 → **Table Editor**에서 테이블 확인 → **RLS**(행 단위 보안) 설정 후 Project URL·anon key 확보

**입력 보관**: `project/info.md` (Supabase URL, anon key)

**디자이너에게 할 말 (6번 완료 후·7번 안내 시)**  
다음은 **7번 Supabase 세팅**이에요. 순서만 따라 하시면 돼요.  
1) **Supabase** 대시보드(https://supabase.com/dashboard)에 들어가서 **New project**로 프로젝트를 하나 만든 뒤,  
2) 왼쪽 메뉴에서 **SQL Editor**를 누르고, **project/db-design.md** 파일을 열어서 **## Supabase DDL** 아래에 있는 **\`\`\`sql\`\`\`** 로 감싸진 부분만 복사해 SQL Editor에 붙여넣은 다음 **Run** 버튼을 눌러 주세요. (테이블이 만들어져요.)  
3) 왼쪽 메뉴 **Project Settings**(톱니바퀴) → **API**에서 **Project URL**과 **anon public** 키를 찾을 수 있어요. 그 두 값을 **채팅으로 알려 주시면** 제가 project/info.md에 반영하고, Vercel에 넣으라고 안내할게요.  

**RLS(읽기/쓰기 허용)**: 기획에 **로그인이 없으면** 프론트에서 조회할 때와 크롤/갱신(API)에서 DB에 저장할 때 **같은 anon 키**를 쓰는 경우가 많아요. 이때 "anon 읽기만 허용"으로 두면 **쓰기가 막혀서 크롤 결과가 DB에 안 들어가는** 문제가 생깁니다. 그래서 **project/db-design.md**에는 **anon 읽기 + 쓰기 둘 다 허용**하거나, 해당 테이블만 **RLS를 끄는** DDL을 넣어 두었어요. SQL 실행 시 그 중 하나를 적용하면 됩니다. 로그인 스펙이 있거나 정책이 다르면 "RLS 설정해 줘"라고 하시면 단계별로 안내할게요. **막히는 곳이 있으면 화면 캡처를 보내 주세요.**

## 8. Jules 요청 (구현)

**먼저 할 일: GitHub repo 선택.** Jules에 들어가면 **반드시 이번 프로젝트의 GitHub 레포를 먼저 선택**해야 합니다. **1번에서 신규로 만든 레포**라면 목록에 안 보일 수 있어요. 그때는 **Configure repo access** → GitHub에서 해당 레포 접근 허용 → Save → Jules로 돌아와 **Choose a repo**에서 이번 프로젝트 레포를 선택한 뒤, 그다음에 프롬프트·시안을 보내면 됩니다.

**복붙할 블록**은 **project/jules-prompt.md**에 있다. 제가 PRD와 기본 포맷을 머징해 둔 파일이므로, 디자이너는 **repo 선택 후** 그 파일에서 블록만 복사해 Jules에 붙이면 된다. **Jules 사용 방법**은 디자이너가 "다음 액션 알려줘" 등으로 물으면 제가 **채팅에서** 안내한다.

**어디선가 막히거나 에러가 나면** 그 화면 캡처나 에러 메시지를 **채팅에 보내 주세요.** 제가 다음에 뭘 누르면 되는지 안내합니다.  
- "Jules 수정 제안(PR)이 안 올라와요" / "레포가 안 보여요" → Configure repo access에서 GitHub 레포 추가 후 Jules에서 다시 **Choose a repo**로 선택  
- "Preview 링크가 안 보여요" → Jules PR 페이지나 Vercel 배포 탭에서 "Preview" / "Visit" 링크 확인

## 9. 피드백·수정 요청 (Preview 확인 시)

Jules가 수정 제안(PR)을 올린 뒤, Preview URL(수정본 미리보기 링크)로 확인하고 **수정하고 싶은 것이 있으면 저에게 알려주세요.**

- **"수정하고 싶은 것이 있으면 저에게 알려주세요!"**라고 안내하고,
- 디자이너와 **충분히 논의한 후**,
- 수정 요청 내용을 **project/jules-prompt.md**에 반영(수정용 블록 추가). **블록 맨 위에 추가 시점을 `YYYY.MM.DD HH:mm` 형식으로 적어 두기.** 논의가 끝나면 **같은 복사할 내용을 채팅에도 한 번 붙여 넣어** "아래 내용을 복사해서 Jules 채팅에 붙여넣으세요"라고 안내. (디자이너는 파일을 열지 않고 채팅에서 바로 복사할 수 있음.)
- 디자이너는 **그 복사할 내용 전체를** Jules에 붙여넣으면 Jules가 수정 반영합니다.

참고: 수정 요청은 `[페이지] + [무엇을] + [어떻게 바꿔]` 포맷으로 정리하면 명확함. 예: `/ 카드 간격을 16→20으로 넓혀줘`

**9번에서 할 일**: 수정 요청을 반복하면서 Preview로 확인해 보시면 됩니다. **더 수정할 게 없고 만족스러우면** 그때 **10번 운영·배포로 가시면 됩니다.** 완성도 있게 마무리한 뒤 10번 운영·배포로 가시면 됩니다.

## (선택) 데이터 소스 — 외부 API

화면에 보여줄 데이터가 **Supabase가 아니라 외부 API**에서 올 때만 정리합니다.

- **무슨 데이터를 보여주나요?** (예: 국가별 날씨)
- **사용할 API/서비스 이름** (모르면 "OO API 추천해줘"라고만 적어도 됨)
- **API 키 발급 경로** (비우면 제가 안내)
- **환경변수 이름** (비우면 제가 제안)
- **API 호출 최소화**: 위 "원칙"에 따라, **캐시 전략**(조건·결과 DB 저장 + 유효 시간)을 PRD·구현에 포함한다.
- **API가 없을 때**: 합법적인 **크롤링**을 고려할 수 있다. 크롤링 시 **주기**(일/주/월 등)를 정해 최신성을 유지한다.

**입력 보관**: `project/required-info.md` 또는 별도 `project/data-sources.md`

## AI 안내 (이 파일 읽었을 때)

1. **`project/state.md`**를 읽어서 현재 **stage**와 **next**를 확인한다. **state.md는 매 단계 완료 시** AI가 갱신한다. stage를 바꾼 뒤에는 **next**를 그 단계에 맞는 다음 액션 한 문장(디자이너에게 할 말)으로 갱신한다.
2. flow-guide는 수정하지 않고, 모든 입력·산출은 **project/** 아래 파일에만 반영한다.
3. 단계별 안내:
   - **stage 1**: "먼저 1번을 진행해주세요. Next.js Boilerplate Deploy → GitHub 연동 → 배포. 끝나면 프로젝트명과 URL을 알려주세요."
   - **stage 2**: "프로젝트명, GitHub URL, Vercel URL을 알려주세요." → `project/info.md` 갱신
   - **stage 3**: "프로젝트 기획서를 전달해 주세요. 간단한 화면이면 한두 줄 설명만이라도 좋아요. 화면을 개발해야 하니 Figma나 Stitch 시안은 필수예요. 전달해 주시면 제가 상세 스펙을 정리해 드릴게요." → `project/screens/` 저장, README 갱신
   - **stage 4**: 먼저 "**기획서와 시안을 모두 제공해 주셨을까요? 모두 제공해 주셨다면 제가 기획서와 시안을 기반으로 계획을 세워 드리겠습니다.**"라고 안내. 제공 완료 시 기획서·시안 기반으로 `project/PRD.md` 생성. **PRD 작성 규칙**: (1) 시니어 기획자 관점 — 요구사항·화면·예외·우선순위·사용자 메시지 정리. (2) 시니어 개발자 관점 — 데이터가 외부 API·크롤링이면 **웹 검색 등으로 리서치** 후, 데이터 전략에 **후보 소스 2~3개 구체 명시**, **각 옵션 난이도·리스크·이유**, **권장안과 이유**, **크롤링 어렵다면 왜 어려운지** 명시. **외부 API/캐시** "원칙 — 외부 API 호출 최소화" 반영(DB 저장 + TTL). 생성 후 "**project/PRD.md를 한 번 확인해 보시고, 수정하거나 더 넣고 싶은 게 있으면 말해 주세요. 괜찮으시면 다음은 5번 필요한 정보 수집이에요.**" 안내. 질문·수정 받고 반영 후 디자이너가 **OK** 하면 `project/state.md`를 5로 갱신
   - **stage 5**: 프로젝트에 필요한 추가 정보(키, 서비스명·요금 등)를 **채팅으로** 디자이너에게 요청. 디자이너가 채팅에서 답하면 AI가 **`project/required-info.md`에 직접 반영**. 디자이너에게 파일을 열어서 채우라고 하지 않음. 다 채우면 6으로
   - **stage 6**: 시안·PRD 기반으로 DDL 생성 → `project/db-design.md` 갱신
   - **stage 7**: 위 7번 **"디자이너에게 할 말"** 문단을 채팅으로 전달. (Supabase 프로젝트 생성 → SQL Editor에서 db-design.md SQL 실행 → Project Settings > API에서 URL·anon key 알려 달라고 안내.)
   - **stage 8**: **project/jules-prompt.md**에 넣는 구현 규칙에는 **로깅(필수, 빡세게)** 요구사항을 반드시 넣는다. 크롤링(요청 시작/종료·수집 행 수·0이면 명시·에러), DB(INSERT/UPDATE 전후·영향 행 수·SELECT 결과 수·에러), API Route(호출 시점·응답 코드·반환 개수/에러), 에러 시 스택 또는 console.error로 원인 추적 가능하게. empty view·에러 원인 추적을 위해 다음 프로젝트에서도 동일하게 적용한다. Jules 사용 방법을 **채팅에서** 안내할 때 **반드시 먼저 GitHub repo 선택**을 강조한다. 순서: (1) Jules 접속·로그인 (2) **Choose a repo**에서 이번 프로젝트 레포 선택 — **신규 레포면 목록에 안 보이므로** Configure repo access → GitHub에서 Repository access에 해당 레포 추가 → Save → Jules로 돌아와 다시 repo 선택 (3) 새 작업 시작 (4) **project/jules-prompt.md** 블록 전체 복사해 붙여넣기 (5) project/screens/ 시안 이미지 첨부 (6) 전송. Supabase는 Vercel에 이미 있으면 그대로 둠. PR 오면 9번 포맷으로 피드백.
   - **stage 9**: Preview URL 확인 후 **"수정하고 싶은 것이 있으면 저에게 알려주세요!"**라고 안내. "더 수정할 게 없고 만족하시면 그때 10번 운영·배포로 가시면 됩니다"라고도 전달. 디자이너가 말하면 논의한 뒤, 수정 요청 내용을 **project/jules-prompt.md**에 반영(수정용 블록 추가 또는 갱신). 추가 시 블록 맨 위에 **현재 시간 `YYYY.MM.DD HH:mm`** 기록. **반영 후에는 반드시** 같은 블록을 **채팅에도 한 번 붙여 넣어** "아래 블록을 복사해서 Jules 채팅에 붙여넣으세요"라고 안내. (디자이너는 파일 없이 채팅에서 바로 복사 가능.) 만족하면 10번 운영·배포로.
   - **stage 10**: Merge, 운영 안내

## 용어 한줄 정리

| 용어 | 한 줄 설명 |
|------|------------|
| **PR** | 수정 제안서. 코드/디자인 변경 내용을 담은 요청. |
| **Merge** | 그 제안을 반영하기(합치기). |
| **Preview URL** | 수정본 미리보기 주소. 배포 전에 확인하는 링크. |
| **RLS** | DB 읽기/쓰기 허용 설정(행 단위 보안). |
| **DDL** | 테이블 구조를 만드는 SQL 문장. |
| **SQL Editor** | Supabase에서 SQL을 붙여넣고 실행하는 화면. |
